# 任务调度与执行

任务调度与执行是 ETL-GO 的核心功能，支持定时任务、手动任务和一次性任务的调度执行。系统内置了强大的调度引擎，基于 cron 表达式提供灵活的调度策略。

## 调度概述

### 调度类型

| 调度类型 | 描述 | 适用场景 |
|----------|------|----------|
| **定时调度** | 按照 cron 表达式定期执行 | 数据同步、报表生成、数据清理 |
| **手动执行** | 用户手动触发执行 | 测试、紧急处理、一次性任务 |
| **立即执行** | 创建后立即执行一次 | 快速验证、实时处理 |
| **依赖调度** | 基于其他任务状态调度 | 工作流、链式任务 |

### 调度状态

| 状态 | 描述 | 可执行操作 |
|------|------|------------|
| **已停止** (0) | 任务未启动调度 | 启动调度、立即执行 |
| **调度中** (1) | 任务正在按计划执行 | 停止调度、手动执行 |
| **执行中** (2) | 任务正在运行 | 查看进度、强制停止 |
| **已暂停** (3) | 任务暂停调度 | 恢复调度、手动执行 |

## Cron 表达式

ETL-GO 使用标准 cron 表达式进行任务调度。

### 基本格式
```
*    *    *    *    *    *
┬    ┬    ┬    ┬    ┬    ┬
│    │    │    │    │    │
│    │    │    │    │    └── 星期几 (0 - 6) (0 = 星期日)
│    │    │    │    └───── 月份 (1 - 12)
│    │    │    └────────── 日期 (1 - 31)
│    │    └─────────────── 小时 (0 - 23)
│    └──────────────────── 分钟 (0 - 59)
└───────────────────────── 秒 (0 - 59)
```

### 特殊值
- `manual` - 手动执行模式
- `*` - 任意值
- `,` - 值列表分隔符
- `-` - 范围
- `/` - 步长
- `?` - 不指定（仅用于日期和星期）

### 常用示例

#### 数据同步场景
```yaml
# 每小时同步一次
cron: "0 0 * * * *"  # 每小时0分0秒

# 每天凌晨2点同步
cron: "0 0 2 * * *"  # 每天2:00:00

# 每30分钟同步一次
cron: "0 */30 * * * *"  # 每30分钟0秒

# 工作日每天9点执行
cron: "0 0 9 * * 1-5"  # 周一至周五9:00:00
```

#### 报表生成场景
```yaml
# 每天凌晨生成日报
cron: "0 0 1 * * *"  # 每天1:00:00

# 每周一凌晨生成周报
cron: "0 0 2 * * 1"  # 每周一2:00:00

# 每月1号凌晨生成月报
cron: "0 0 3 1 * *"  # 每月1号3:00:00

# 每季度第一天凌晨生成季报
cron: "0 0 4 1 1,4,7,10 *"  # 1月、4月、7月、10月1号4:00:00
```

#### 数据清理场景
```yaml
# 每天凌晨清理7天前数据
cron: "0 0 3 * * *"  # 每天3:00:00

# 周末凌晨深度清理
cron: "0 0 4 * * 0,6"  # 周六、周日4:00:00

# 每小时清理临时数据
cron: "0 0 * * * *"  # 每小时0分0秒
```

## 任务配置

### 基本配置
```yaml
任务配置:
  名称: "用户数据同步"
  描述: "每小时同步用户数据到数据仓库"
  
  # 调度配置
  调度:
    类型: "cron"           # cron: 定时调度, manual: 手动
    表达式: "0 0 * * * *"  # 每小时执行一次
    时区: "Asia/Shanghai"  # 时区设置
    
  # 执行配置
  执行:
    超时时间: "3600s"      # 任务执行超时时间
    重试次数: "3"         # 失败重试次数
    重试间隔: "60s"       # 重试间隔时间
    并发限制: "1"         # 并发执行数限制
```

### 高级配置
```yaml
高级调度配置:
  # 调度策略
  错过执行处理: "立即执行"  # 立即执行/跳过/下次执行
  重叠执行处理: "等待"      # 等待/并行执行/跳过
  
  # 资源限制
  内存限制: "512MB"        # 最大内存使用
  CPU限制: "1"            # CPU核心数限制
  
  # 环境变量
  环境变量:
    - key: "ENV"
      value: "production"
    - key: "LOG_LEVEL"
      value: "info"
  
  # 依赖配置
  依赖任务:                # 依赖的其他任务
    - "数据准备任务"
    - "变量计算任务"
  
  # 触发器
  触发器:
    - 类型: "文件变化"
      参数:
        - key: "监控目录"
          value: "/data/input"
        - key: "文件模式"
          value: "*.csv"
        - key: "变化类型"
          value: "新增"
```

## 故障排除

### 常见问题

#### 1. 调度不执行
**问题**: 任务配置了调度但从未执行
**可能原因**:
- cron 表达式错误
- 任务状态不是"调度中"
- 调度器未启动
- 系统时间不正确

**解决方案**:
```yaml
检查步骤:
  1. 验证cron表达式: echo "0 0 * * * *" | cron-parser
  2. 检查任务状态: GET /api/task/{id}
  3. 检查调度器状态: GET /api/system/scheduler
  4. 验证系统时间: date && ntpq -p
```

#### 2. 调度时间不准确
**问题**: 任务执行时间与预期不符
**可能原因**:
- 时区设置错误
- 系统负载过高
- 调度器延迟
- cron解析问题

**解决方案**:
```yaml
调整步骤:
  1. 设置明确时区: 时区: "Asia/Shanghai"
  2. 检查系统负载: top, vmstat
  3. 优化调度时间: 避开高峰时段
  4. 使用更简单cron表达式
```

#### 3. 并发执行冲突
**问题**: 同一任务多个实例同时执行
**可能原因**:
- 任务执行时间过长
- 调度器配置问题
- 分布式环境冲突
- 锁机制失效

**解决方案**:
```yaml
解决措施:
  1. 设置任务并发限制: 并发限制: "1"
  2. 增加执行超时时间: 超时时间: "1800s"
  3. 检查锁机制: 分布式锁配置
  4. 优化任务执行时间
```

#### 4. 调度器崩溃
**问题**: 调度器进程异常退出
**可能原因**:
- 内存泄漏
- 死锁
- 资源耗尽
- 代码缺陷

**解决方案**:
```yaml
恢复步骤:
  1. 重启调度器服务
  2. 检查错误日志: /var/log/etl-go/scheduler.log
  3. 监控资源使用
  4. 升级到最新版本
```

## 注意事项

1. **时间精度**: cron表达式支持秒级精度，但实际执行可能有几秒延迟
2. **时区设置**: 确保调度器和应用使用相同时区
3. **系统负载**: 避免在系统高峰时段调度重要任务
4. **资源竞争**: 合理设置并发限制，避免资源竞争
5. **故障恢复**: 设计任务时考虑幂等性和可恢复性
6. **监控告警**: 配置完善的监控和告警机制
7. **版本兼容**: 升级时注意调度器的版本兼容性
8. **数据一致性**: 分布式环境下注意数据一致性问题
